- **2026-07-07** : **Correction de l'entrée du 2026-07-06 ci-dessus — la piste "423 / session cookie" du 2026-06-26 n'était PAS un faux positif.** Elle avait simplement besoin de conditions réalistes pour se reproduire : un serveur MCP persistant (`DEVSYS_vsp_test`), avec de nombreuses requêtes stateless (lectures) et stateful (lock/update) partageant le même `cookiejar` avant une séquence Lock→Update. Les tests `go test` isolés (client/jar neufs à chaque test) ne la reproduisaient jamais, d'où la conclusion erronée du 2026-07-06. En testant en réel sur `/ZDEMO/ISU_GROUPING_JOB1_M2` via le vrai outil MCP `EditSource`, le 423 `InvalidLockHandle` est réapparu deux fois de suite (lock handles différents à chaque fois).
    - **Root cause confirmée** : le `cookiejar` standard de Go (`pkg/adt/config.go`, `NewHTTPClient()`) n'a aucune notion de session ADT "stateful" vs "stateless" — avec beaucoup de trafic mixte sur un client longue durée, il peut présenter un `sap-contextid`/`SAP_SESSIONID` obsolète à une requête stateful (Lock/Update), qui atterrit sur une autre instance de serveur applicatif SAP que celle détenant le verrou.
    - **Fix** : réintroduction propre (sans logging de cookies sur disque, contrairement au hack abandonné le 2026-06-26) d'un pinning de session dans `pkg/adt/http.go` — `sessionPinningRoundTripper` : capture le nom/valeur du cookie de session (`extractSessionID`, désormais `(name, value string)`) sur chaque réponse, et le réattache uniquement aux requêtes marquées `X-sap-adt-sessiontype: stateful`, en le retirant des requêtes stateless.
    - **Régression auto-introduite puis corrigée dans la foulée** : `Activate` (`pkg/adt/devtools.go`) ne positionnait PAS `Stateful: true` dans ses `RequestOptions`. Avant le fix de pinning, cela n'avait pas d'importance car le cookiejar envoyait de toute façon tous les cookies sur toutes les requêtes. Une fois le pinning en place, `Activate` s'est retrouvé traité comme "stateless" et donc envoyé sans le cookie de session pinné — atterrissant potentiellement sur un autre serveur applicatif que celui du Lock+Update précédent. Conséquence observée en réel : `EditSource` répondait `success: true` avec une activation "réussie" (corps de réponse vide, interprété par `parseActivationResult` comme un succès total), **sans que le contenu ne soit réellement modifié sur le serveur** — vérifié par relecture (`GetInclude`) montrant l'ancien contenu inchangé, de façon stable et reproductible (pas un simple délai de réplication). Fix : ajout de `Stateful: true` à l'appel d'activation dans `Activate()`.

    - **Bug #3 (`SyntaxCheck` silencieux sur un INCLUDE nu)** : en tentant de relire une erreur volontairement introduite par l'utilisateur dans `/ZDEMO/ISU_GROUPING_JOB1_M2`, `SyntaxCheck` retournait un résultat vide (`[]`) au lieu du texte d'erreur — donnant l'illusion trompeuse d'un code propre. Root cause (confirmée via dump XML brut du `checkrun`) : SAP ne peut pas sémantiquement vérifier un `INCLUDE` seul, sans contexte de programme principal, et répond `chkrun:status="notProcessed"` / `chkrun:statusText="No main program for include ... found"` avec une `checkMessageList` vide — ce n'est PAS un bug de ce codebase mais une limitation de l'API ADT elle-même. Confirmé transposable : lecture réussie d'une vraie erreur de syntaxe sur la classe `/ZDEMO/CL_DEMO_CORE` (`Type "/ZDEMO/ISU_GRP_ID2" is unknown.`), donc `SyntaxCheck` fonctionne normalement dès qu'un contexte complet existe (classe, programme).
      - **Fix (choix utilisateur, option B)** : ajout de la constante `SyntaxCheckNotProcessed` dans `pkg/adt/devtools.go` ; `parseSyntaxCheckResults` détecte `report.Status != "processed"` et remonte un résultat explicite `Severity: "notProcessed"` avec le `StatusText` réel de SAP, au lieu d'une liste vide silencieuse. Dans `pkg/adt/workflows_edit.go`, la boucle de traitement des résultats de `EditSourceWithOptions` ignore ce cas particulier (ni erreur ni warning, `continue`) pour ne pas bloquer/perturber le workflow d'édition d'includes déjà validé — l'outil `SyntaxCheck` autonome, lui, remonte l'info telle quelle.
      - **Vérifié en conditions réelles** (délégué à un agent MCP en lecture seule) : `SyntaxCheck` sur `/sap/bc/adt/programs/includes/%2FZDEMO%2FISU_GROUPING_JOB1_M2` retourne bien `[{"severity":"notProcessed","text":"No main program for include /ZDEMO/ISU_GROUPING_JOB1_M2 found", ...}]` — comportement conforme au fix, plus de faux négatif silencieux.
      - **Limite non résolue** : lire le texte réel d'une erreur de syntaxe *à l'intérieur* d'un INCLUDE nu reste impossible via `SyntaxCheck` seul — nécessiterait de vérifier l'include dans le contexte de son programme principal (non implémenté, piste future si besoin).
