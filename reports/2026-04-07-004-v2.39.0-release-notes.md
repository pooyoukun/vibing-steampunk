# VSP v2.39.0 Release Notes

**Date:** 2026-04-07
**Report ID:** 004
**Subject:** v2.39.0 "Transport Changelog & Change Correlation" Release

---

## Executive Summary

VSP v2.39.0 adds two thin but important CLI analysis slices on top of SAP CTS data:

- `vsp changelog <package>` for package-level transport history
- `vsp changes <package>` for CR-level grouping via request attributes from `E070A`

This release matters because VSP now answers two different questions cleanly:

- "Which transport requests changed this package?"
- "Which transport requests belong to the same logical change?"

That second question is the bridge from raw transport history to real change-correlation across workbench and customizing requests.

---

## Highlights

### Package Changelog

- `vsp changelog '$ZDEV'`
- `vsp changelog '$ZDEV' --since 20260101`
- `vsp changelog '$ZDEV' --include-subpackages --format json`

What it does:

- resolves package scope once
- finds package objects through TADIR/package acquisition helpers
- reads transport-object membership from `E071`
- collapses task -> parent request via `E070-STRKORR`
- loads request header data from `E070`
- loads request descriptions from `E07T`
- outputs grouped transport history in text or JSON

Why it matters:

- gives a practical package changelog without opening SE09/SE10
- works for package-level analysis instead of single-object revision browsing
- is a reusable base for higher-level change grouping

### CR-Level Change Grouping

- `vsp changes '$ZDEV' --attribute SAPTEST`
- `vsp changes '$ZDEV' --since 20260101 --format json`
- `vsp changes '$ZDEV' --include-subpackages`

What it does:

- reuses the transport history gathered for the package
- loads request attributes from `E070A`
- groups requests by attribute reference value
- leaves requests without that attribute in an explicit `(untagged)` bucket

Why it matters:

- supports logical change grouping above raw transports
- allows correlation of workbench and customizing requests when they share a CTS attribute
- creates the first product surface for CR-level co-change analysis

## Configuration

New CLI configuration for transport attribute selection:

- `.vsp.json` per system:
  - `transport_attribute`
- env vars:
  - `VSP_TRANSPORT_ATTRIBUTE`
  - `VSP_<SYSTEM>_TRANSPORT_ATTRIBUTE`

Typical example:

```json
{
  "default": "a4h-105",
  "systems": {
    "a4h-105": {
      "url": "https://example.sap",
      "user": "DEVELOPER",
      "client": "001",
      "transport_attribute": "SAPTEST"
    }
  }
}
```

This keeps correlation logic out of code and lets each landscape pick its own attribute naming convention.

## Important Fixes

### E070 vs E07T for Transport Descriptions

`AS4TEXT` does not live in `E070`.

The `changelog` slice originally queried it from `E070`, which fails on packages with real transport history. The correct implementation is:

- request header fields from `E070`
- request description text from `E07T`

This fix is now part of the shipped slice and was confirmed against a real SAP system.

### Correct Total Transport Counting in `vsp changes`

The initial summary counted `groups + untagged`, which undercounted real transport totals whenever:

- multiple transports belonged to one group
- one transport had multiple attribute values

The final version stores `TotalTransports = len(entries)` and reports the real transport count.
