# VSP v2.38.0 Release Notes

**Date:** 2026-04-06
**Report ID:** 012
**Subject:** v2.38.0 "Graph Intelligence & Safe Refactoring Previews" Release

---

## Executive Summary

VSP v2.38.0 is a real minor release.

This slice adds a coherent analysis layer on top of the existing CLI and MCP foundation:

- graph intelligence for co-change, impact, and config usage
- real usage-example extraction with snippets
- package and object health snapshots
- standard API surface inventory
- first safe refactoring-preview tools

The release is important because these commands change the shape of the product.
VSP is no longer only "execute/query/read ABAP from the terminal".
It now has a distinct static-analysis and change-safety surface for both humans and AI coding assistants.

---

## Highlights

### Graph Intelligence

- `vsp graph co-change <type> <name>`
- `vsp graph where-used-config <variable>`
- MCP `impact` with optional `include_source_analysis=true`
- Mermaid and HTML export for graph CLI flows

What this adds:

- transport-history co-change analysis
- static reverse impact analysis with parser augmentation
- first-pass config reader discovery for `TVARVC`
- browser-friendly graph export without SVG/rendering dependencies

### Real Usage Examples

- `vsp examples FUNC Z_FM`
- `vsp examples CLAS ZCL_FOO --method BAR`
- `vsp examples PROG ZREP --submit`
- `vsp examples PROG ZREP --form BUILD`
- MCP `usage_examples`

What this adds:

- real call-site snippets instead of only reference lists
- stronger grounding for AI coding assistants
- faster API onboarding for unfamiliar custom code

### Health and Surface Inventory

- `vsp health --package '$ZDEV'`
- `vsp health CLAS ZCL_FOO`
- `vsp api-surface '$ZDEV'`

What this adds:

- health snapshots from ATC, tests, boundaries, and staleness
- package/object quality checks before edits
- inventory of top-used SAP standard APIs from custom code

### Safe Refactoring Previews

- `vsp slim '$ZDEV'`
- `vsp rename-preview CLAS ZCL_OLD ZCL_NEW`
- `vsp class-sections ZCL_FOO`
- `vsp method-signature ZCL_FOO GET_DATA`

What this adds:

- dead-code candidate discovery
- rename blast-radius preview with risk warnings
- structured class/member inspection
- targeted method contract reading without scanning the whole class

---

## Important Notes

- `impact` is strong for static reverse dependencies, but it is not runtime truth.
- `where-used-config` is currently the clearer config-reader tool; config-aware `impact` is still only partial.
- `slim` v1 reports dead-code candidates from static indexes; dynamic/framework entrypoints can still exist.
- `class-sections` and `method-signature` are useful helpers, but the stronger long-term direction is a unified code-unit contract model.
